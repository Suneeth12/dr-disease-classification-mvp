# Retina Model Review

Retina Model Review is a diabetic retinopathy review app with:

- a FastAPI backend for model loading, inference, case history, and Grad-CAM artifacts;
- a React/Vite frontend for uploading retinal images and reviewing results;
- local-only runtime artifacts in `runtime/`.

The repository is GitHub-ready: source code, tests, and lightweight runtime JSON files are tracked, while trained model weights, uploads, databases, generated outputs, dependency folders, and caches are ignored.

## 1. Prerequisites

Install these before running the project:

- Python 3.12.x. Python 3.12.10 was used during verification.
- Node.js 22.13 or newer, or Node.js 24 or newer.
- Git.
- Enough disk space for TensorFlow dependencies and model weights.

Do not create the backend virtual environment with Python 3.14. The backend pins
`tensorflow==2.20.0`, and TensorFlow did not provide a matching wheel for Python
3.14 in this setup. Python 3.12 is the safest supported version for this project.
If another Python version is installed, use the `py -3.12` commands below so
Windows does not select the wrong interpreter.

Check installed Python versions:

```powershell
py -0p
```

Use PowerShell for the commands below.

## 2. Get The Code

If you cloned from GitHub:

```powershell
cd retina-model-review
```

If you are using an already-downloaded project folder, open PowerShell in that folder:

```powershell
cd retina-model-review
```

## 3. Download Runtime Model Files

Model weights are not committed to GitHub because they are large. Everything else needed by the app,
including the E-REG recipe and threshold JSON files, is kept in GitHub. Download only the model ZIP
from this Google Drive folder:

[Download model/runtime artifacts from Google Drive](https://drive.google.com/drive/folders/1OsLOgkkSU_drRmFESKNo6d0V2WPbBGlZ?usp=sharing)

Open the link in your browser, download the shared folder contents, and unzip them locally if Google Drive gives you a ZIP file.

Best Google Drive layout:

```text
Google Drive folder/
  retina-models.zip
```

Best ZIP contents:

```text
  models/
```

Keep the Drive folder focused on model weights only. Do not put source code, `node_modules`, `.venv`,
uploads, app databases, notebook JSON artifacts, or generated Grad-CAM outputs in Drive. For the
easiest setup, upload one ZIP named `retina-models.zip` with `models/` at the top level, so users can
unzip it directly into `runtime/`.

After unzipping the model ZIP, the local layout should look like this:

```text
runtime/
  models/
    attention.keras
    lesion.keras
    multiscale.keras
    patch_mil.keras
    multibranch.keras              # optional; keep it if included in the downloaded bundle
  notebook-artifacts/
    ensemble/
      final_ensemble_recipe.json        # included current E-REG recipe
    thresholds/
      attention_thresholds.json         # included
      lesion_thresholds.json            # included
      multiscale_thresholds.json        # included
      patch_mil_thresholds.json         # included
  data/
    uploads/                       # generated by local runs
    artifacts/                     # generated by local runs
    app.db                         # generated by local runs
```

Create the expected folders first:

```powershell
New-Item -ItemType Directory -Force runtime\models
New-Item -ItemType Directory -Force runtime\data\uploads
New-Item -ItemType Directory -Force runtime\data\artifacts
```

If the Google Drive ZIP contains a `models` or `Models` folder, unzip or copy its `.keras` files into
`runtime\models`. Do not replace `runtime\notebook-artifacts` during normal setup; those lightweight
recipe and threshold files come from GitHub.

The included E-REG recipe expects these model files:

```text
runtime\models\attention.keras
runtime\models\patch_mil.keras
runtime\models\lesion.keras
runtime\models\multiscale.keras
```

The backend loads model files by recipe name. If `final_ensemble_recipe.json` lists a model named `attention`, the matching local file must be `runtime\models\attention.keras`.

## 4. Backend Setup

From the repository root:

```powershell
cd backend
py -3.12 -m venv .venv
.\.venv\Scripts\python.exe -m pip install --upgrade pip
.\.venv\Scripts\python.exe -m pip install -r requirements.txt
.\.venv\Scripts\python.exe -m pip install --no-deps -e .
```

Confirm the virtual environment is using Python 3.12 before installing
requirements:

```powershell
.\.venv\Scripts\python.exe --version
```

Expected output should start with:

```text
Python 3.12
```

If it shows Python 3.14 or another unexpected version, delete `.venv` and recreate
it with `py -3.12 -m venv .venv`.

The final `pip install --no-deps -e .` command installs the local `src/retina_api`
package into the virtual environment. Without it, `uvicorn` cannot import
`retina_api.app`.

Start the backend:

```powershell
.\.venv\Scripts\python.exe -m uvicorn retina_api.app:app --host 0.0.0.0 --port 8000
```

The API runs at:

- Backend: `http://localhost:8000`
- API docs: `http://localhost:8000/docs`
- Health endpoint: `http://localhost:8000/api/v1/health`

The first startup can take time because TensorFlow and the model runtime load in the background. Wait until the health endpoint reports `ready: true` before running real predictions.

## 5. Frontend Setup

Open a second PowerShell window from the repository root:

```powershell
cd frontend
npm install
npm run dev
```

The Vite dev server prints a local URL, usually:

```text
http://localhost:5173
```

The frontend uses `http://localhost:8000` as the default backend URL. To override it:

```powershell
Copy-Item .env.example .env
notepad .env
```

Set:

```text
VITE_API_BASE_URL=http://localhost:8000
```

## 6. Run The App

Use two terminals:

1. Backend terminal, from the repository root:

```powershell
cd backend
.\.venv\Scripts\python.exe -m uvicorn retina_api.app:app --host 0.0.0.0 --port 8000
```

2. Frontend terminal, from the repository root:

```powershell
cd frontend
npm run dev
```

Then open the Vite URL in your browser, upload a PNG/JPG/JPEG retinal image, and wait for the backend runtime status to become ready.

## 8. Runtime Environment Variables

The default runtime layout is under `runtime/`. You can override paths with environment variables before starting the backend:

```powershell
$env:RETINA_RUNTIME_DIR="..\runtime"
$env:RETINA_DATA_DIR="..\runtime\data"
$env:RETINA_MODELS_DIR="..\runtime\models"
$env:RETINA_NOTEBOOK_ARTIFACTS_DIR="..\runtime\notebook-artifacts"
$env:RETINA_DATABASE_URL="sqlite:///../runtime/data/app.db"
```

These relative examples assume the backend is started from `backend/`, as shown in the commands above. Most users should not set these variables at all; the defaults already point to the repository `runtime/` folder.

## 9. Troubleshooting

- Backend says models are missing: check `runtime\models\<model-name>.keras` and the model names in `runtime\notebook-artifacts\ensemble\final_ensemble_recipe.json`.
- Backend says thresholds are missing: check `runtime\notebook-artifacts\thresholds\*_thresholds.json`.
- Frontend cannot connect: confirm the backend is running at `http://localhost:8000` and `VITE_API_BASE_URL` matches it.
- Health stays in loading: give TensorFlow time to load, and check the backend terminal for missing artifact errors.
- `pip install -r requirements.txt` says no matching TensorFlow distribution: delete `.venv`, install/use Python 3.12.x, then recreate the venv with `py -3.12 -m venv .venv`.
- `ModuleNotFoundError: No module named 'retina_api'`: from `backend/`, run `.\.venv\Scripts\python.exe -m pip install --no-deps -e .`, then start `uvicorn` again.
- `npm install` shows audit warnings: run `npm audit` to inspect them. Do not commit `node_modules`.
- `npm run build` has Node engine warnings: use Node.js 22.13 or newer, or Node.js 24 or newer.

## 10. GitHub Upload Guidance

Before pushing:

1. Run `git status --short`.
2. Confirm `.keras` files, uploads, app databases, ZIP archives, `node_modules`, and build output are not staged.
3. Commit source code, tests, and tracked lightweight runtime JSON files.
4. Keep model weights and private datasets in Google Drive or another approved external storage location.
